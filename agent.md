## 注释规则归口

仓库内源码必须补充中文注释；没有中文注释、注释过少或注释已经过期的代码，应在相关变更中同步补齐。该要求作为当前仓库的显式工程约定执行，并与 `AGENTS.md`、各模块 README 和 `docs/standards/development.md` 共同生效。

### 总体要求

- `main` 目录下源码必须具备中文注释，说明类型职责、模块位置、输入输出边界和关键安全约束。
- `test` 目录下源码必须具备中文注释，说明测试目的、覆盖场景、构造数据和预期结果。
- 新增代码在提交前必须一并补齐中文注释，不允许长期留下“先写功能、后补注释”的缺口。
- 修改旧代码时，如发现注释缺失、过期或与实现不一致，必须同步修正。

### 注释质量

- 注释必须服务于理解设计和实现，重点解释“为什么存在、解决什么问题、边界在哪里”。
- 禁止只写“给变量赋值”“调用某方法”这类机械性空话。
- 优先使用 Javadoc 或 JSDoc 描述类型、组件、公开方法和测试意图；复杂代码块前可以补充少量行内中文注释。
- 注释必须与当前实现保持一致，禁止复制模板后不更新。

### 存量代码治理

- 存量源码的中文注释缺口需要逐步补齐，不得无限期拖延。
- 当前优先覆盖控制面、认证鉴权、策略审计、工作流、执行 Worker、前端 API 边界和安全关键测试。

## 任务进度维护要求

- 所有研发任务在进入新阶段、完成关键交付或收敛剩余风险后，都必须及时更新开发任务清单进度。
- 至少同步更新与当前工作对应的事实源，例如项目计划、任务排期表、模块化任务清单或其他正式进度文档。
- 进度更新必须反映真实完成度、已完成事项和剩余条件，禁止只改百分比不改事实说明。

## 结果导向执行约定

- 在不触碰 `AGENTS.md` 安全红线、模块边界和事实源优先级的前提下，Agent 默认采用以结果为导向的执行方式。
- 对于可以通过仓库上下文、测试、契约和现有实现自行判定的技术细节，Agent 应直接实现、验证并收口，不重复向用户请求过程确认。
- 只有当任务将违反安全基线、改变产品范围、需要新的架构决策，或遇到外部不可消除阻塞时，Agent 才应暂停并升级说明。
- Agent 在推进过程中必须同步补齐测试、文档、进度和验证证据，不能只完成局部代码修改就宣称任务完成。
