每个 SQL 会话的三种模式

SQL 编辑、自然语言生成和 Compare 都属于同一个 SQL tab；自然语言先生成可审查草稿，用户确认后再应用到编辑器。

SQL 1 会话
SQL | 自然语言 | Compare
自然语言模式
连接：h2-local-test
库名：ST1
表名：ELADREF
相关字段：CONFIG_KEY, CONFIG_VALUE, CONFIG_GROUP



用户描述：查询配置组 A 中启用的配置项

生成 SQL 草稿：select CONFIG_KEY, CONFIG_VALUE from ST1.ELADREF where CONFIG_GROUP = 'A' and ENABLED = 'Y'
假设：ENABLED 字段表示启用状态；CONFIG_GROUP 是配置组字段。假设会随 SQL 一起展示。
“引用当前编辑器 SQL”默认关闭；打开后，模型会把当前编辑器内容当作修改基准，适合“加条件”“改汇总”“调整排序”等场景。
用户点击“应用到编辑器并校验”后，草稿才会写入 SQL 编辑器并触发服务端校验；当前编辑器内容不会被自动覆盖。
服务端校验失败时：展示真实错误、错误位置和只读校验原因；用户点击“让 AI 修正”后，模型基于错误信息生成新版草稿。

自然语言模式第一版

  • 只生成 SELECT。
  • 生成后先显示在自然语言结果区，不自动覆盖 SQL 编辑器。
  • 用户点击“应用到编辑器并校验”后才写入当前 SQL 会话。
  • 用户提供连接、库名、表名和字段提示。
  • 当前编辑器 SQL 只在用户打开开关后作为上下文。
  • 校验失败先展示真实错误，AI 修正必须由用户点击触发。

不做

  • 不生成 INSERT / UPDATE / DELETE。
  • 不依赖尚未接入的对象元数据自动推断表结构。
  • 不让模型绕过服务端校验和人工执行确认。
  • 不在失败时自动循环修 SQL。
