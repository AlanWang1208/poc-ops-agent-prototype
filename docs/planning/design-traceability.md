# 设计范围追溯清单

本文件用于约束项目规划必须严格遵循《企业级智能运维 Agent 开放底座架构设计蓝图 V8.1》。

## 产品边界

- 系统为公司内部自研自用。
- 系统为单组织部署。
- 设计中没有租户或多租户概念。
- 不规划租户标识、租户隔离、租户路由、租户计费和租户管理。
- 用户权限边界使用设计中的用户、角色、Skill Category、参数权限、目标环境和审批机制表达。

## 为实现设计目标而补充的工程机制

以下内容不是新增产品能力，而是为了让设计中的安全、恢复、审计和生产运行目标能够落地：

- 持久化工作流作为执行事实源。
- 幂等键、补偿和人工接管。
- Skill 签名、灰度和回滚。
- AgentScope 文件系统 Skill 与平台契约分离：`backend/skills/<skill>/SKILL.md` 面向 Agent，`backend/contracts/skills/packages/<skill>/` 面向注册、签名、Schema 和测试。
- 动态模型供应方设置用于替代静态环境变量配置：只支持单组织内部管理员维护 OpenAI-compatible 供应方，API Key 加密保存并只返回指纹，测试配置只返回脱敏连通性状态，不新增租户、计费、外部客户接入或模型市场能力。
- 强类型 CommandEnvelope 和版本化跨模块契约。
- 受限 Worker、短期凭据和受控网络出口。
- 测试评测、安全验证、故障注入和发布门禁。
- 在没有外部 IdP 的部署中，由 M01 实现正式内建身份域、账号会话和受控密码重置，但不得改变单组织产品边界。
- M09 操作台首轮重写采用 JavaScript / JSX、JSDoc、TypeScript `checkJs`、Zod、Vitest 和 Playwright，属于前端工程落地机制，不新增产品边界。
- SQL 工作台在 P1 提供连接目录、SQL 校验、开发/测试环境受控单条 `SELECT` 执行和 DML 静态预检报告；任何开发环境写执行仍按 P2 受控变更试点追溯到策略、工作流、Worker 和审计链。

这些机制不得改变产品边界。需要新增产品能力时，必须先获得用户确认并更新设计。

## 禁止擅自引入的能力

- 多租户和商业化计费。
- 面向外部客户的开放平台能力。
- 未经审批的生产写操作。
- 绕过环境策略、分析快照、持久化工作流或审计链的开发测试写操作。
- 任意脚本生成后直接执行。
- 由人格或 Prompt 降低组织安全要求。
- 与设计目标无关的技术平台化扩展。
- 把 `local-oidc` / Mock OIDC 直接当成生产身份方案。

## 规划变更检查

每次新增模块、任务、契约或基础设施前，回答：

1. 它对应 design 的哪一章节或目标？
2. 它是产品能力，还是实现该产品能力的工程机制？
3. 它是否改变自研自用、单组织部署的产品边界？
4. 它是否引入 design 未定义的新角色、新权限或新业务流程？
5. 如果改变产品边界，是否已经获得用户确认？
