# ADR 0006：内部 Team Workspace 逻辑租户边界

- 状态：Accepted
- 日期：2026-06-13
- 负责人：架构负责人
- 相关模块：M01、M02、M03、M04、M05、M08、M09、M10、M11
- 相关任务：Team Workspace 定制闭环

## 背景

项目原始边界是公司内部自研自用、单组织部署，并明确不引入租户概念。现在产品需求变更为：同一公司内部的多个团队需要在共享底座上进行自由定制，包括各自启用的 Skill、角色映射、目标环境和策略限制。

如果继续维持完全单组织模型，后续不同团队的 Skill 可见性、审批要求、审计范围和工作流恢复会混在同一安全上下文中，无法满足团队级自治和隔离要求。

## 决策

引入内部 `Team Workspace` 作为平台的逻辑租户边界。

1. `Team Workspace` 仅表示公司内部团队工作空间，不表示外部 SaaS 租户。
2. 首期采用共享数据库中的逻辑隔离，核心事实表必须携带 `workspace_id`。
3. 一个用户可以加入多个 Team Workspace，并在请求中选择当前工作空间。
4. 角色授权从账号全局角色升级为工作空间作用域角色；平台级管理员角色可以作为例外，但必须进入审计。
5. 工作空间可以配置启用的 Skill、目标环境、参数约束和额外策略限制。
6. 平台全局安全基线先于工作空间策略执行；工作空间策略只能等价或收紧，不得放宽平台基线。
7. 命令、Worker 请求、语义事件、工作流、幂等键和审计事件必须携带工作空间上下文。

## 明确不包含

- 外部客户接入。
- 商业化租户、套餐和计费。
- 每工作空间独立部署。
- 每工作空间独立数据库或独立 Schema。
- 由工作空间配置降低平台安全基线。
- 操作台在浏览器中自行做授权决策。

## 影响

正面影响：

- 多团队可以共享底座并进行受控定制。
- Skill 路由、策略、审计和工作流恢复具备团队级边界。
- 后续扩展审批、目标系统连接和制品保留策略时有稳定作用域。

负面影响：

- 身份、策略、契约、工作流和前端都需要增加工作空间上下文。
- 现有 v1 契约不包含工作空间，需要新增 v2 契约。
- 测试必须覆盖跨工作空间隔离和默认拒绝路径。

## 验证方式

- 契约测试覆盖 workspace-aware v2 Schema。
- 身份测试覆盖一个用户多个工作空间和工作空间角色。
- 策略测试覆盖平台基线不可被工作空间放宽。
- 工作流测试覆盖幂等键加入 `workspace_id` 后跨工作空间不冲突。
- 前端测试覆盖工作空间选择后只提交当前工作空间上下文。

## 发布与回滚

当前仍处于 POC 阶段，无生产历史数据迁移。实施时新增 v2 契约并保留 v1 历史契约；如果 Team Workspace 未通过验收，可在 P1 范围内回退到单默认工作空间，但不得回退为无审计、无策略或无隔离的路径。
