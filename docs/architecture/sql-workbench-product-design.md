# SQL 工作台产品设计

## 1. 产品目标

SQL 工作台面向内部研发工程师，目标是在一个受控界面内完成开发与测试环境的日常 SQL 分析、查询和变更，减少在 AI 分析工具与传统数据库客户端之间切换。

工作台复用现有身份、策略、工作流、Worker、语义事件和审计链路，不新增部署服务或模块编号。生产 SQL 连接始终不可见、不可调用。

## 2. 分阶段能力

| 能力 | P1 | P2 目标态 |
|---|---|---|
| 开发环境 `SELECT` | 受控执行 | 受控执行 |
| 开发环境 `INSERT`、`UPDATE`、`DELETE` | 仅预检 | 分析后受控执行 |
| 测试环境 `SELECT` | 受控执行 | 受控执行 |
| 测试环境 `INSERT`、`UPDATE`、`DELETE` | 仅预检 | 按风险策略执行、二次确认或审批 |
| 生产环境 | 连接不可见、不可调用 | 连接不可见、不可调用 |
| DDL、`CALL`、存储过程、`MERGE` | 禁止 | 默认禁止，后续必须通过独立 ADR 决策 |

P2 开发环境 DML 不是绕过安全链路的直接 JDBC 写入。每次写操作必须绑定精确 SQL、参数哈希、连接、环境、策略版本和分析报告，并通过持久化工作流提交给受限 Worker。

## 3. 核心交互

SQL 工作台首先提供与主流数据库客户端一致的开发体验，大模型能力作为增强层接入，不改变研发人员熟悉的操作路径。

### 3.1 标准数据库工作台

- 左侧对象浏览器展示当前授权连接下的 Schema、表、视图、索引和字段。
- 对象浏览器支持独立向左收起和恢复；收起后编辑器与结果区使用释放出的横向空间。
- 默认 Schema 不在顶部工具栏重复展示；用户可从对象浏览器选择 Schema 并将其设为当前 SQL 文件的默认上下文。
- 中间使用多 SQL 文件标签和传统文本编辑器，支持保存、格式化、执行当前语句、执行脚本、停止和 Explain。
- 编辑器根据 AST 识别光标所在的完整 SQL 语句，而不是机械执行单行文本。
- 下方结果区提供结果表、消息、执行计划、历史、导出、分页和执行耗时。
- P1 的 `INSERT`、`UPDATE`、`DELETE` 执行动作只触发预检。
- P2 开发环境的低风险 DML 可进入分析后执行；中高风险 DML进入确认、审批或拒绝。
- SQL、参数、环境或 Schema 变化后，已有分析和授权状态立即失效。

### 3.2 AI SQL 助手

- AI 助手位于编辑器右侧，并采用横向折叠抽屉。收起时从左向右缩入右侧窄边栏，编辑器与结果区同步向右扩展，实际释放横向空间。
- AI 对当前语句提供错误分析、性能优化、SQL 解释和结果异常检查。
- 错误与性能问题应关联到具体 SQL 片段，并允许用户查看建议修改或优化前后对比。
- AI 建议必须由用户主动采纳，并重新进入服务端校验与策略链。

### 3.3 统一工作区专注模式

- Agent 工作区、RAG 问答和 SQL 工作台使用一致的“展开工作区”入口。
- 展开后，左侧主菜单向左收起，当前页面右侧辅助卡片向右收起，中央工作区使用释放出的横向空间。
- SQL 工作台进入专注模式时，同时收起数据库对象浏览器和右侧 AI SQL 助手，使编辑器与查询结果占满工作区。
- 再次点击“退出展开”恢复原布局，不改变当前会话、SQL 文件、查询结果或页面上下文。

## 4. 会话与事务模型

- “工作台会话”表示一个用户可恢复的 SQL 文件标签集合，不等同于长期数据库连接。
- 每个 SQL 文件标签可以绑定连接和 Schema，但默认使用短生命周期 JDBC 连接。
- `SELECT` 使用独立只读请求，不共享事务。
- P2 DML 使用单次短事务：执行成功自动提交，失败自动回滚。
- 默认不提供跨 SQL 单元的手动 `Commit`、`Rollback` 或长期事务。
- 如果未来确需交互式事务，必须单独评审锁风险、空闲超时、强制回滚、恢复和审计语义。

## 5. AI 分析边界

- 大模型用于解释 SQL、发现潜在业务风险、生成优化建议和辅助构造预检摘要。
- 大模型输出始终是不可信建议，不能授权执行，也不能降低服务端策略。
- 服务端 AST、对象权限、环境策略、风险规则和数据库权限是强制边界。
- Copilot 只接收授权后的脱敏元数据、脱敏 SQL、查询计划和预检摘要。
- AI 分析失败时，安全规则仍必须独立拒绝危险或越权 SQL。

## 6. 受控 DML 数据流

```text
研发人员选择语义动作
  -> 服务端 AST 与对象权限校验
  -> DML 预检与 AI 风险分析
  -> M02 根据环境、角色、风险和影响范围决策
  -> M05 持久化工作流绑定 SQL 与分析快照
  -> M07 Worker 再次校验并在短事务中执行
  -> 自动提交或自动回滚
  -> M09 展示强类型结果与语义事件
  -> M02 / M10 写入审计与可观测证据
```

## 7. 风险分级建议

- 低风险：开发环境、允许 Schema、带精确条件、影响行数低于策略阈值，可由授权研发人员确认后执行。
- 中风险：影响范围较大、涉及敏感字段或测试环境写入，需要增强确认或审批。
- 高风险：无 `WHERE`、无法可靠预检、越权对象、危险函数或超出影响阈值，直接拒绝。
- 生产环境：连接不可见并在 API、策略和 Worker 三层拒绝。

## 8. 验收重点

- 研发人员可使用与主流数据库客户端一致的对象浏览、编辑、执行和结果查看流程完成开发环境受控增删改查。
- P1 中所有 DML 执行动作只能触发预检，不存在写执行路径。
- P2 中开发环境 DML 必须经过分析、策略、持久化工作流、短事务和审计链。
- 多 SQL 文件标签可保存独立上下文，只读查询可并行运行和独立取消。
- SQL 或参数修改后，旧分析报告、确认和审批自动失效。
- 生产连接无法通过界面、API、控制面配置或 Worker 路径访问。
