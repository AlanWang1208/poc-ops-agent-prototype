# SQL 工作台会话增强设计

## 1. 背景与目标

本设计固化 2026-06-30 已确认的 SQL 工作台增强范围，归属 M09 SQL 工作台 P1 切片，并衔接 M02 策略审计、M04 模型交互、M05 工作流事实源、M07 受限执行 Worker 和 M11 测试验证。

目标是让每个 SQL 会话同时支持三类工作方式：

1. SQL 编辑模式：继续作为用户查看、编辑、校验和执行只读 SELECT 的主入口。
2. 自然语言模式：根据用户描述生成 SELECT 草稿，帮助用户编写 SQL，但不自动执行。
3. Compare 模式：在受控只读边界内，对两个对比目标的结果做数据一致性对比，并提供确定性 diff 与 AI 摘要。

同时补充单会话 `.sql` 导入导出能力，用于把当前 SQL 会话内容导入或保存到本地文件。

## 2. 非目标

- 不生成、执行或绕过限制执行 `INSERT`、`UPDATE`、`DELETE`、`MERGE`、DDL、存储过程或任意脚本。
- 不开放生产写连接，不改变 P1 只读边界。
- 不把 AI 输出作为授权、执行事实或一致性结论来源。
- 不做跨数据源泛化对比；第一版聚焦同一个连接下不同库名或 schema 的同表对比。
- 不做大表离线核对、全量历史报表持久化或长期 diff 存储。
- 不依赖尚未接入的对象元数据自动推断表结构。
- 不新增租户、计费、外部客户接入或模型市场能力。

## 3. 会话模型与界面结构

每个 SQL 会话保留一个独立的会话上下文，包含：

- 当前 SQL 编辑器内容。
- SQL 校验报告与最近一次执行结果。
- 自然语言生成表单、生成草稿、假设列表和修正记录。
- Compare 表单、生成 SQL、确定性 diff 报告和 AI 摘要。
- 导入文件名和导出建议文件名。

会话内提供 `SQL / 自然语言 / Compare` 三个模式切换。模式切换不丢失当前会话中的编辑器、草稿、报告和结果状态。SQL 编辑器仍是唯一执行入口；自然语言和 Compare 都只能生成、校验或触发受控只读对比流程。

## 4. `.sql` 导入导出

第一版采用浏览器本地文件 API，不新增后端上传或文件存储接口。

导入流程：

1. 用户点击导入，选择本地 `.sql` 文件。
2. 前端校验扩展名为 `.sql`。
3. 如果当前 SQL 编辑器已有内容，弹出覆盖确认。
4. 用户确认后，将文件内容写入当前会话的 SQL 编辑器。
5. 导入不会自动校验或执行。

导出流程：

1. 用户点击导出。
2. 前端提示用户输入文件名。
3. 用户取消或输入空文件名时终止导出。
4. 文件名缺少 `.sql` 后缀时自动补齐。
5. 前端使用 Blob 下载当前 SQL 编辑器内容。

导入导出只影响当前 SQL 会话，不创建后端制品，不写审计中的文件内容。后续如需要服务端文件库，必须另行设计权限、审计和存储边界。

## 5. 自然语言模式

### 5.1 输入

自然语言模式第一版由用户显式提供上下文：

- 当前连接。
- 库名或 schema。
- 表名。
- 相关字段提示。
- 用户自然语言描述。
- 是否引用当前 SQL 编辑器内容。

“引用当前 SQL 编辑器内容”默认关闭。打开后，模型把当前编辑器 SQL 作为修改基准，适合“加条件”“改成汇总”“调整排序”等场景。关闭时，模型只基于用户填写的结构化提示和自然语言描述生成新 SELECT 草稿。

### 5.2 生成与应用

自然语言模式只生成 SELECT 草稿：

1. 用户点击生成。
2. 控制面调用受控模型能力，要求返回结构化结果：`draftSql`、`assumptions`、`clarifyingQuestion`、`refusalReason`。
3. 服务端把模型输出当作不可信数据，先做语法、单语句和只读分类校验。
4. 生成结果显示在自然语言结果区，不自动覆盖 SQL 编辑器。
5. 用户点击“应用到编辑器并校验”后，草稿才写入当前 SQL 编辑器，并触发现有服务端校验。
6. 执行仍必须由用户回到 SQL 编辑模式或使用会话内执行按钮明确触发。

模型信息不足时必须返回澄清问题或假设列表，不得伪造表结构。用户要求生成 DML、DDL、脚本或生产写操作时，服务端返回拒绝结果。

### 5.3 校验失败与 AI 修正

如果应用后的服务端校验失败：

1. 界面展示真实校验错误、错误位置、只读校验原因和可审计错误码。
2. 用户可以点击“让 AI 修正”。
3. 控制面把原始草稿、用户描述、用户提供的结构化提示和安全脱敏后的校验错误提交给模型。
4. 模型返回新版草稿和修正说明。
5. 新版草稿仍停留在自然语言结果区，用户再次点击“应用到编辑器并校验”后才写入编辑器。

系统不自动循环修 SQL，避免模型在用户不可见的情况下反复改变查询。

## 6. Compare 模式

### 6.1 对比目标

第一版 Compare 面向 AS/400 常见场景：同一个数据源连接下，通过不同库名区分环境或配置集合，例如 `ST1.ELADREF` 与 `ST2.ELADREF`。

用户填写：

- 连接。
- 基准库名。
- 对比库名。
- 表名。
- 输出字段。
- 业务 key 字段。
- 可选 where 条件片段。
- 忽略字段。
- `maxRows` 上限。

第一版要求两侧表名相同、字段同名、字段含义一致。字段映射、跨连接对比和跨数据库方言转换作为后续扩展。

### 6.2 SQL 生成与校验

Compare 不接受两段任意自由 SQL。后端根据结构化输入生成两条 SELECT：

```sql
select <fields> from <baseLibrary>.<tableName> where <condition>
select <fields> from <compareLibrary>.<tableName> where <condition>
```

where 输入只允许表达条件本身，不允许分号、多语句、`order by`、`group by`、`limit`、`union` 或非条件子句。库名、表名、字段名和 key 字段都按服务端标识符规则校验。生成后的两条 SQL 必须分别通过现有 SQL AST 校验和只读边界校验。

界面默认折叠生成 SQL，用户可以展开查看和复制，但不能通过编辑生成 SQL 来绕过 Compare 表单约束。

### 6.3 执行与 diff

Compare 流程由控制面发起：

1. M09 后端接收 Compare 请求并做输入校验。
2. M02 执行策略授权和审计记录。
3. 控制面生成两条只读 SELECT，并通过 M07 SQL Worker 路径执行。
4. Worker 仍执行自己的只读拒绝边界，不信任控制面生成的 SQL。
5. 控制面拿到两侧结果后计算确定性 diff。
6. 前端展示 diff 报告、样例差异和可选 AI 摘要。

diff 以业务 key 为基准归并行：

- 基准侧存在、对比侧缺失。
- 对比侧存在、基准侧缺失。
- key 相同但非忽略字段值不同。
- 两侧都存在且所有参与字段一致。

报告包含行数、匹配 key 数、缺失行数、差异行数、差异字段数、样例差异和 `maxRows` 截断提示。如果任一侧达到 `maxRows`，报告标记为受限结果，不给出绝对一致结论。

### 6.4 失败与降级

如果一侧 SQL 执行失败，Compare 状态为 `INCOMPLETE`：

- 展示成功侧的基础摘要。
- 展示失败侧的安全错误信息。
- 不输出“一致”或“不一致”结论。
- AI 摘要只能解释失败和下一步排查建议，不能补造缺失结果。

如果 AI 摘要失败，确定性 diff 仍完整展示，页面标记“AI 摘要不可用”。Compare 不因为模型失败而失败。

## 7. AI Compare 摘要

AI Compare 摘要只基于确定性 diff 事实生成，不能读取完整结果集，不能重新执行 SQL，不能覆盖 diff 结论。

提交给模型的内容限制为：

- Compare 请求摘要。
- 两侧目标名。
- 行数和差异统计。
- 有限数量的脱敏差异样例。
- 失败或截断状态。

模型输出包含：

- 一句话结论摘要。
- 主要差异类型。
- 需要人工关注的 key 或字段样例。
- 后续排查建议。

第一版优先复用 SQL 工作台现有 AI SQL 助手能力，增加 `COMPARE_SUMMARY` 类动作。只有当 Compare 摘要需要被 AgentScope 主链路复用时，才新增只读 Skill 契约，例如 `sql-compare-summary-read`。该 Skill 也只能消费已生成的 CompareReport，不得执行 SQL。

## 8. 安全、审计与隐私

- 模型输出、用户输入、SQL 草稿、Compare 表单和 Worker 结果全部视为不可信数据。
- 自然语言模式和 Compare 模式都不能直接执行模型生成内容。
- 所有可执行 SQL 必须经过服务端 AST 校验、只读分类、M02 授权和 M07 Worker 二次校验。
- 前端不得根据展示文本推断授权状态，仍通过 API 返回的强类型状态渲染。
- Prompt 中不得包含密钥、完整连接凭据或生产敏感数据。
- Compare AI 摘要只接收统计信息和有限脱敏样例。
- 审计记录应包含操作人、连接、目标环境、动作类型、请求摘要、策略版本、traceId、requestId 和执行结果，不记录模型 API Key 明文。
- `.sql` 导入导出不上传文件内容；仅在用户显式校验或执行后进入现有 SQL 审计链路。

## 9. 错误处理

自然语言模式需要覆盖：

- 模型供应方未配置或凭据拒绝。
- 用户请求超出 SELECT 范围。
- 信息不足，需要用户补充库、表或字段。
- 模型返回无法解析或无法通过只读校验的 SQL。
- “引用当前编辑器 SQL”打开时，当前 SQL 本身不合法。

Compare 模式需要覆盖：

- 标识符非法。
- where 条件非法或包含多语句。
- key 字段为空或不在输出字段中。
- 两侧查询任一侧校验失败。
- 两侧查询任一侧执行失败。
- 结果达到 `maxRows` 上限。
- AI 摘要不可用。

所有错误都应给出稳定错误码、面向用户的中文解释和面向开发者的可审计诊断字段。

## 10. 测试与验收

前端测试：

- 单会话 `.sql` 导入、覆盖确认、取消覆盖、导出文件名补全。
- `SQL / 自然语言 / Compare` 模式切换不丢状态。
- 自然语言草稿不自动覆盖编辑器。
- “应用到编辑器并校验”和“让 AI 修正”交互。
- Compare 表单校验、生成 SQL 折叠展示、diff 报告和 AI 摘要降级。

后端测试：

- 自然语言生成接口拒绝 DML、DDL、多语句和模型异常输出。
- 引用当前 SQL 开关只在显式打开时传入编辑器 SQL。
- Compare 输入校验覆盖非法标识符、非法 where、空 key、超大 `maxRows`。
- Compare 两条生成 SQL 均通过只读校验后才提交 Worker。
- 一侧失败、AI 摘要失败和 `maxRows` 截断状态可预测。
- 审计事件不包含密钥或模型凭据明文。

安全与评测：

- Prompt 注入要求执行写操作时必须拒绝。
- 模型把 DML 包装成注释、CTE 或多语句时必须拒绝。
- Compare where 条件注入 `; drop table`、`union`、`order by` 时必须拒绝。
- AI 摘要不得声称执行 SQL 或替代确定性 diff 结论。

浏览器验收：

- 1280px、1440px、1920px 桌面视口下三模式可用。
- 展开工作区后 SQL 编辑、自然语言结果区和 Compare 报告不互相遮挡。
- 错误提示同时包含用户可读解释和开发者可用诊断字段。

## 11. 发布与回滚

建议按能力分阶段启用：

1. 启用 `.sql` 导入导出，仅前端本地能力，无后端依赖。
2. 启用自然语言生成接口，但默认不自动执行，模型未配置时失败关闭。
3. 启用 Compare 确定性 diff，AI 摘要作为可降级能力。

回滚方式：

- 前端隐藏自然语言和 Compare 模式入口，不影响 SQL 编辑器既有校验与执行。
- 后端关闭自然语言生成和 Compare 摘要动作后，SQL 工作台仍可执行受控 SELECT。
- Compare API 如关闭，不影响已有 SQL 校验和单条执行路径。

## 12. 后续扩展

后续可在独立设计中评估：

- 跨连接或跨数据库的数据一致性对比。
- 字段映射和类型转换规则。
- 大表异步对账和短期报告存储。
- 复用 CompareReport 的只读 Agent Skill。
- 接入对象浏览器元数据后，辅助推荐字段、key 和 where 条件。
